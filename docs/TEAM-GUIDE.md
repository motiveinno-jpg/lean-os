### 3-4. 권한·다계정 격리 (두 분 공동)

> **가장 중요**. RLS 104테이블이 실제로 막는지 확인.

1. 본부장이 두 번째 테스트 회사 생성 (`테스트컴퍼니B`)

3. **회계담당자 계정으로 로그인** → 두번째 회사의 데이터가 **절대 보이면 안 됨**

   - `https://www.owner-view.com/deals/{B사 딜ID}` → 403 또는 빈 화면
   - `https://www.owner-view.com/partners/{B사 거래처ID}` → 403
5. 회계담당자 역할을 `staff`로 낮추고, 관리자 전용 메뉴 접근 시도:
   - /settings/billing → 막혀야 함
   - /settings/team → 읽기만 되고 수정 막혀야 함
   - /admin/* → 완전 차단

**한 건이라도 뚫리면 즉시 `critical` 라벨로 이슈 등록.**

## 7. 버그 리포트 양식 (필수 준수)

### 제목 포맷
`[심각도][모듈] 한줄요약` 예: `[🔴][세금계산서] 공급가액 자동계산 -1원 오차`

### 본문 템플릿
```markdown
## 환경
- 계정: junhoyeon55@gmail.com (본부장)
- 역할: owner
- 브라우저/기기: Chrome 125 / Mac, iPhone 15 Safari 등
- 발생 시각: 2026-04-17 14:32
- URL: https://www.owner-view.com/tax-invoices/...

## 재현 절차
1. ...
2. ...
3. ...

## 기대 동작
(어떻게 돼야 하는가)

## 실제 동작
(어떻게 됐는가 — 에러 메시지 전문 포함)

## 스크린샷 / 영상
(첨부)

## 심각도
- [ ] 🔴 Critical: 서비스 사용 불가 · 데이터 유실 · 보안 · 결제 오류
- [ ] 🟡 High: 핵심 기능 오작동이지만 우회 가능
- [ ] 🟢 Medium: 불편하지만 사용 가능
- [ ] 🔵 Low: 개선 제안

## 영향받는 사용자 범위
- [ ] 모든 사용자
- [ ] 특정 역할만 (어느 역할?)
- [ ] 모바일만 / 데스크톱만
- [ ] 특정 브라우저만
```

### 라벨 사용 규칙
- `critical` · `high` · `medium` · `low` (하나 필수)
- `bug` · `ux` · `feature` · `question` (하나 필수)
- `mobile` · `accounting` · `sales` · `admin` (해당 시)

### 이슈 끝나면
- Claude가 수정 완료 → 해당 이슈에 커밋 SHA + 배포시각 댓글
- 보고자가 **같은 시나리오 재검증** → `verified` 댓글 + close

## 10. 자주 묻는 질문

**Q. 테스트 중 데이터를 다 지우고 처음부터 하고 싶어요**
A. /settings/company > 위험구역 > "회사 초기화" (테스트 회사에서만). 실 회사에선 금지.

**Q. 본부장과 회계담당자가 같은 회사에서 테스트할 때 충돌나요**
A. §3-4 역할 분리를 지키고, 같은 레코드를 동시에 편집하지 마세요. E6 엣지케이스가 바로 그 검증입니다.

**Q. 에러가 났는데 Sentry에 자동으로 가나요?**
A. 네. 단, 보고서에 URL+시각을 꼭 적어야 Claude가 Sentry에서 찾을 수 있습니다.

**Q. 스테이징 환경이 따로 있나요?**
A. 없습니다. www.owner-view.com = 프로덕션. 대신 회사 단위 RLS로 격리돼 있어 두 분 테스트가 다른 회사에 영향 안 줍니다.

**Q. 대표님(CEO)은 이 과정에 어떻게 참여하시나요?**
A. 주 1회 월요일 오전 진행 리뷰만. 긴급 의사결정 필요 시 `critical` 이슈 + 카톡.
