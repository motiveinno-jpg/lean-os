# OwnerView — 프로젝트 규칙

## 🚨 MOTIVE Brain (모든 AI·터미널 강제 — 최우선 규칙)

**이 섹션은 Claude / Codex / Cursor / Cline / Aider / 기타 모든 AI 에이전트에게 적용된다.**

### 세션 시작 시 — 첫 응답 전에 무조건 실행
1. **`cat ~/motive-brain/state/ownerview.md` 전문 읽기** (915줄 / 111KB, 토큰 비용 무시하고 풀로)
2. **`tail -80 ~/motive-brain/lessons.md` 읽기** (최근 교훈)
3. 위 2개 안 읽고 코드 변경 시작 = 룰 위반. 첫 응답에서 brain 상태 1줄 요약 후 작업 진행.

### Claude Code 자동화
- `.claude/settings.json` 의 SessionStart hook 이 자동 `cat` 실행 → 컨텍스트에 강제 주입.
- Codex CLI 는 `AGENTS.md` 최상단 동일 지시 — 세션 시작 시 brain 읽기 강제.
- Cursor 는 `.cursorrules` 동일 지시.

### 작업 중 / 종료 시
- **중요 결정**: `~/motive-brain/decisions.md` append (수기, 자동 아님)
- **실패/성공 교훈**: `~/motive-brain/lessons.md` append (인시던트 발생 시 즉시)
- **세션 종료 전**: `~/motive-brain/state/ownerview.md` 변경분 갱신 (현재 진행·완료·BLOCKED 토픽)
- **잊지 말 것**: 이 규칙은 수동이 아니다. brain 미갱신 = 다음 세션 컨텍스트 손실 = 인시던트 재발.

### 사용자 명시 요청 (2026-05-21)
> "응 자동으로 매번 읽게해 너 뿐만 아니라 다른 터미널들이랑 ai들이 읽어야돼"
→ SessionStart hook + AGENTS.md + .cursorrules 3중 강제. 토큰 비용보다 정합성 우선.

## 🤖 멀티 에이전트 라우팅 — **메인 직접이 기본값**

이 프로젝트는 `.claude/agents/` 에 도메인별 10개 서브에이전트를 보유한다. 그러나 **기본 동작은 "메인 Claude가 직접 작업"**이다. 에이전트 위임은 비용이 크다 — 스폰마다 콜드스타트(CLAUDE.md + 자기 정의 재독 + **코드베이스 0부터 재탐색** 5~15콜)가 붙고, 게이트 체이닝(도메인→qa→security)은 **직렬**이라 벽시계 시간을 몇 배로 늘린다. 토큰 카운터가 낮아 보여도 실제 작업이 서브에이전트 안에 숨어 직렬로 도는 것일 뿐이다.

**원칙: 에이전트는 아래 L 트리거가 명확할 때만 쓴다. 그 외에는 메인이 직접 한다.** 라우팅 판단 자체로 시간을 쓰지 말 것 — 단일 도메인이면 판단 없이 바로 직접 작업 착수.

### 🚦 작업 티어링 (S가 기본 — L 트리거 없으면 무조건 S)

먼저 **"L 트리거가 있나?"만 본다.** 없으면 티어 판정에 더 시간 쓰지 말고 S로 직접 착수.

| 티어 | 판정 기준 | 처리 방식 | 검증 |
|---|---|---|---|
| **S (기본값 — 작업 대부분)** | 단일 도메인의 거의 모든 작업: 오타·라벨·CSS·버그픽스·기능 추가/수정·리팩터. **여러 파일이어도 한 도메인 안이면 S.** DB·RLS·시크릿·결제 무관 | **메인이 직접 처리. 에이전트 0개. 코드베이스 재탐색 없이 바로 편집.** | 메인이 `tsc --noEmit` (변경 크면 `npm run lint` 추가). 빌드 실패는 Vercel이 잡으므로 풀빌드 생략 |
| **M (예외 — 드묾)** | 단일 도메인이지만 메인이 그 도메인 로직 컨텍스트가 **부족하다고 명시적으로 판단**될 때만. 단순 모름·귀찮음은 사유 아님 | 도메인 에이전트 1개. worktree 안 씀 | `qa-validator` **fast 모드** (tsc+lint+grep). security 생략 |
| **L (에이전트가 정당한 유일한 경우)** | (1) **3개+ 독립 도메인 동시** 변경 / (2) **DB 스키마·RLS·마이그레이션** / (3) **결제(Stripe·Toss)·시크릿·권한** 변경 | 도메인 병렬 + `isolation: "worktree"` (DB 포함 시 `db-architect` 선행) | `qa-validator` **full 모드** (풀빌드+Playwright) → `security-reviewer` 필수 |

- 에이전트를 쓸 때(M·L)는 **정확한 파일 경로·함수명을 프롬프트에 명시**해 탐색 콜드스타트를 줄인다 ("X 고쳐줘" ✗ → "`src/app/bank/page.tsx`의 `sortTx`를 …" ✓).
- **애매하면 위가 아니라 아래(S)로.** 에이전트는 손해가 기본값이므로, L 트리거 3종이 명백할 때만 올린다. 2개 도메인은 메인이 직접(S) 처리.
- 단일 도메인 작업을 "안전하게" M으로 올리지 말 것 — 이게 과거 지연의 주범이었다.

**DB 변경 포함 시**: 티어 무관 `db-architect` 먼저 → 마이그레이션 적용·타입 재생성 → 결과를 도메인 에이전트에 전달 (이 경우 자동 L 티어).

**Notion QA 후처리** (코드 푸시 완료 후, S/M/L 공통): 메인이 commit hash + 변경 파일 + 화면위치를 정리해 `notion-agent` 호출 → 완료 토글 + 진행요약 + 작업로그 갱신. **단, 세션 중 여러 건 푸시 시 건건이 부르지 말고 세션 끝에 1회 배치 호출** (notion-agent 콜드스타트 절감). Notion 단독 요청은 코드 무관 즉시 `notion-agent`.

### Worktree 격리
- **L 티어(다중 도메인 병렬)에서만** `Agent` 도구의 `isolation: "worktree"` 사용. S·M 티어는 worktree 금지 (생성/정리 + 컨텍스트 재구성 오버헤드).
- 변경 없으면 worktree 자동 정리, 변경 있으면 메인이 받아서 머지

### BLOCKED 영역 (모든 에이전트 금지)
- 홈택스 sync (`tax-invoices/`, `codef-sync` 안의 hometax/cash-receipt 분기)
- CF-12200, CF-00007, CF-00000 에러 우회 시도
- 사유: CODEF 운영팀 답변 대기 중 (자동 메모리 `project_hometax_blocked.md` 참조)

### 직접 작성이 기본 (= S 티어 = 작업의 대부분)
메인 직접 작업은 예외가 아니라 **기본 경로**다. 아래는 특히 명백한 직접 케이스:
- 단일 도메인의 모든 수정 (오타·라벨·CSS·버그·기능·리팩터, 파일 수 무관)
- 사용자가 "이 파일 직접 봐줘" 명시
- 탐색·조사·질문 응답
- `motive-brain` 상태 갱신 (메인 책임)

에이전트로 위임하는 건 위 L 트리거(다중 도메인 3+·DB 마이그레이션·결제/RLS 보안) 3종이 명백할 때**만**이다.

## 🧠 LLM 코딩 가이드라인 — 흔한 실수 줄이기

LLM이 코딩할 때 흔히 저지르는 실수를 줄이기 위한 행동 규칙. 위 도메인 규칙과 함께 적용.

**트레이드오프**: 이 규칙들은 속도보다 **신중함**에 가중치를 둔다. 사소한 작업에는 판단으로 적용 강도 조절.

### 1. 코딩 전에 생각하라

**가정하지 말 것. 혼란을 숨기지 말 것. 트레이드오프를 드러낼 것.**

구현 전:
- 가정을 **명시**한다. 불확실하면 묻는다.
- 해석이 여러 가지면 **다 제시**한다. 조용히 하나 고르지 말 것.
- 더 단순한 접근이 있으면 말한다. 정당할 땐 사용자 요구를 push back.
- 불명확하면 멈춘다. 무엇이 헷갈리는지 짚고 묻는다.

### 2. 단순함 우선

**문제를 푸는 최소 코드만. 추측성 코드 금지.**

- 요청 범위 **밖** 기능 추가 금지.
- 1회용 코드에 추상화 금지.
- 요청하지 않은 "유연성"·"설정 가능성" 추가 금지.
- 일어날 수 없는 시나리오를 위한 에러 처리 금지.
- 200줄 짠 게 50줄로 가능하면 다시 쓴다.

자문: **"시니어 엔지니어가 보고 '오버엔지니어링이다' 할 것 같나?"** → 그렇다면 단순화.

### 3. 정밀한 변경

**꼭 필요한 곳만 건드린다. 본인이 만든 흔적만 정리한다.**

기존 코드 편집 시:
- 인접한 코드·주석·포맷팅을 함부로 **"개선"하지 말 것**.
- 고장 안 난 것 리팩토링 금지.
- 본인 스타일과 달라도 **기존 스타일에 맞춘다**.
- 무관한 데드 코드를 발견하면 **언급만**, 삭제는 X.

변경으로 고아 코드(orphan)가 생기면:
- 본인 변경으로 unused 된 import/변수/함수 **만** 제거.
- 사전부터 있던 데드 코드는 명시 요청 없으면 제거 금지.

**테스트**: 모든 변경된 라인이 사용자 요청과 직접 연결돼야 한다.

### 4. 목표 기반 실행

**성공 기준을 정의한다. 검증될 때까지 루프한다.**

작업을 **검증 가능한 목표**로 변환:
- "검증 추가" → "잘못된 입력 테스트 작성 후 통과시키기"
- "버그 픽스" → "재현 테스트 작성 후 통과시키기"
- "X 리팩토링" → "전후 테스트 통과 보장"

다중 스텝 작업이면 간단한 계획을 먼저:
```
1. [스텝] → 검증: [체크]
2. [스텝] → 검증: [체크]
3. [스텝] → 검증: [체크]
```

명확한 성공 기준 = 독립적으로 루프 가능. 약한 기준("동작하게") = 매번 사용자 확인 필요.
